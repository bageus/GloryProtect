# GloryProtect — обязательные правила проекта

Этот файл является **единственным источником правды для инженерных правил проекта**. Правила обязательны для разработчиков, автоматических агентов, Codex и любых будущих участников.

Архитектурные границы и владельцы данных определены в [`ARCHITECTURE.md`](ARCHITECTURE.md).

---

## 1. Базовые требования

1. Проект разрабатывается на **Godot 4.6.2 stable** и GDScript.
2. Нельзя создавать монолитные классы, сцены или менеджеры.
3. Каждый файл должен иметь одну ясно сформулированную ответственность.
4. Любая механика реализуется как система или небольшой компонент с определённой границей ответственности.
5. Нельзя отклоняться от [`ARCHITECTURE.md`](ARCHITECTURE.md) без одновременного обновления архитектурного документа.
6. В каждом домене должен существовать только один источник правды для изменяемого состояния.
7. Хардкод должен быть минимизирован или исключён.
8. Любые настраиваемые игровые значения должны находиться в ресурсах данных, а не быть разбросаны по скриптам.

---

## 2. Ограничение размера файлов

### 2.1. Жёсткий лимит

**Ни один исходный файл проекта не может превышать 600 строк.**

Лимит применяется к:

- `.gd`;
- `.cs`, если C# когда-либо появится в проекте;
- `.tscn`, созданным вручную и содержащим значительную логику конфигурации;
- большим JSON, YAML и другим вручную поддерживаемым конфигурациям;
- архитектурным и техническим документам, если они превращаются в неудобный монолит.

В число строк входят:

- код;
- комментарии;
- пустые строки;
- встроенные структуры данных.

### 2.2. Предупредительный порог

При достижении примерно **450 строк** файл необходимо оценить на разделение.

Если файл продолжает расти, до достижения 600 строк он должен быть разделён по ответственности.

### 2.3. Запрещённые обходы

Нельзя формально обходить лимит путём:

- удаления полезных пробелов и комментариев;
- объединения нескольких выражений в одну строку;
- помещения большого количества логики во вложенные классы одного файла;
- переноса монолита в один огромный `.tscn`;
- создания одного файла с множеством несвязанных статических функций.

Цель лимита — модульность, а не уменьшение визуального количества строк.

## 3. Запрет монолитов

### 3.1. Один файл — одна ответственность

Каждый класс должен отвечать за одну область поведения.

Хорошие примеры:

- `WindSystem` — состояние и изменение ветра;
- `PlatformMovementController` — горизонтальное движение платформы;
- `AnchorOverloadController` — перегрузка якорей;
- `AnchorOperationQueue` — очередь операций;
- `ShieldSystem` — прочность пяти секций;
- `CrewRoleManager` — назначения ролей;
- `CombatResolver` — завершение атак и урон.

Плохой пример:

- `GameManager`, который одновременно создаёт врагов, двигает платформу, меняет щит, обновляет UI, выдаёт карточки и управляет паузой.

### 3.2. Когда файл необходимо разделить

Файл должен быть разделён, если выполняется хотя бы одно условие:

- у класса больше одной независимой причины для изменения;
- класс управляет несколькими доменами;
- класс хранит состояние, владельцем которого должна быть другая система;
- класс содержит несколько независимых машин состояний;
- тестирование одной функции требует запуска почти всей игры;
- название класса становится слишком общим: `Manager`, `Controller`, `Utils`, `Helper` без уточнения домена;
- файл приближается к предупредительному порогу размера.

### 3.3. Композиция вместо наследования

Предпочтение отдаётся небольшим компонентам:

```text
Defender
├── HealthComponent
├── MovementComponent
├── RoleAssignmentComponent
├── TargetSelector
├── MeleeAttackComponent
└── DefenderStateMachine
```

Не следует строить глубокие цепочки наследования для вариантов поведения.

## 4. Один источник правды

### 4.1. Правило владения состоянием

Каждое изменяемое состояние имеет ровно одного владельца.

Примеры:

| Состояние | Единственный владелец |
|---|---|
| Текущее состояние забега и паузы | `GameFlowController` |
| Направление и сила ветра | `WindSystem` |
| Горизонтальная скорость платформы | `PlatformMovementController` |
| Состояния четырёх якорей | `AnchorSystem` |
| Прочность пяти секций | `ShieldSystem` |
| Текущая роль защитника | `CrewRoleManager` |
| Здоровье персонажа | его `HealthComponent` |
| Монеты текущего забега | `RunEconomy` |
| Номер и цена следующего улучшения | `UpgradeSystem` |
| Занятость клеток объектами | `BuildableGrid` |

### 4.2. Производные данные

UI, визуальные компоненты и вспомогательные системы могут хранить только временный кэш, который можно полностью восстановить из источника правды.

Нельзя хранить отдельные независимые копии одного состояния, например:

- процент щита одновременно в UI и `ShieldSystem`;
- назначенную роль одновременно в защитнике и панели экипажа;
- состояние якоря одновременно в платформе, UI и враге;
- монеты одновременно в HUD и системе улучшений.

### 4.3. Изменение состояния

Состояние изменяется только через публичный интерфейс его владельца:

```gdscript
shield_system.apply_damage(section_id, amount)
crew_role_manager.request_assignment(defender_id, role_id)
anchor_system.request_toggle(anchor_id)
run_economy.add_coins(amount, source)
```

Другие системы не должны напрямую менять внутренние поля владельца.

## 5. Минимизация хардкода

### 5.1. Что должно находиться в данных

В `.tres`-ресурсы или специализированные конфигурации выносятся:

- силы ветра;
- скорость и ускорение платформы;
- сопротивление и максимальная скорость;
- длина тросов;
- длительность установки, снятия и возврата якоря;
- время перегрузки;
- ширина зон контакта и установки;
- скорость зарядки;
- параметры волн;
- параметры физического спавна;
- здоровье, скорость и кулдауны типов врагов;
- параметры защитников;
- время лечения;
- параметры турелей;
- цены и определения улучшений;
- цвета, пороги и другие UI-настройки.

### 5.2. Магические числа и строки

Запрещены необъяснённые значения внутри игровой логики:

```gdscript
# Плохо
velocity.x += 178.0 * delta
if health <= 25.0:
    show_alert()
```

```gdscript
# Хорошо
velocity.x += movement_balance.steering_force * delta
if health_percent <= shield_balance.critical_threshold:
    show_alert()
```

Повторяемые идентификаторы оформляются через:

- `StringName`;
- enum;
- ресурс определения;
- централизованный каталог идентификаторов, если он действительно необходим.

Пути к узлам не должны многократно повторяться строками по проекту.

### 5.3. Допустимые константы

В коде могут оставаться только структурные инварианты, являющиеся частью правил игры, например:

- количество секций щита — 5;
- количество якорей — 4;
- базовый урон обычного удара — 1;
- количество карточек в выборе — 2.

Даже такие значения следует именовать и документировать, если они используются более одного раза.

## 6. Системная реализация

### 6.1. Любая механика имеет владельца

Перед добавлением функции необходимо определить:

1. к какому домену она относится;
2. какая система владеет её состоянием;
3. какие команды система принимает;
4. какие события система публикует;
5. какие данные необходимы для настройки;
6. как система тестируется отдельно.

Если на эти вопросы нет ответа, реализацию начинать нельзя.

### 6.2. Команды и события

Рекомендуемый поток:

```text
Input/UI
→ команда владельцу системы
→ проверка допустимости
→ изменение состояния
→ сигнал о результате
→ обновление визуала и зависимых систем
```

UI не выполняет последствия сам.

Пример:

```text
AnchorButton
→ request_toggle_anchor(anchor_id)
→ AnchorSystem
→ anchor_state_changed
→ AnchorVisualController + HUD + BoardingPathRegistry
```

### 6.3. Зависимости

Предпочтительный порядок:

1. явная ссылка, переданная родительской сценой;
2. экспортируемый `NodePath` для локальной сцены;
3. сигнал;
4. небольшой интерфейс компонента.

Запрещается строить проект на произвольных глобальных поисках узлов и глубоких путях вида:

```gdscript
get_node("../../../../SomeManager/OtherSystem")
```

Глобальные автозагрузки допускаются только для действительно глобальной инфраструктуры, а не для всей логики забега.

## 7. Соблюдение архитектуры

[`ARCHITECTURE.md`](ARCHITECTURE.md) является единственным источником правды для:

- границ систем;
- владельцев состояния;
- разрешённых направлений зависимостей;
- структуры каталогов;
- основных команд и событий;
- машин состояний;
- правил расширения проекта.

Если новая задача требует изменить архитектуру:

1. сначала определить необходимое изменение;
2. обновить `ARCHITECTURE.md` в том же коммите;
3. только затем менять или добавлять код;
4. проверить отсутствие второго конкурирующего источника правды.

Нельзя незаметно вводить новую глобальную систему, обходя описанную архитектуру.

## 8. Сцены Godot

### 8.1. Сцена — композиция, а не монолит

Большие сцены делятся на самостоятельные переиспользуемые сцены:

- платформа;
- защитник;
- физический враг;
- наземный шар;
- якорь или визуал якоря;
- турель;
- медицинский пост;
- HUD;
- миникарта.

### 8.2. Логика не должна жить только в `.tscn`

Сложное поведение оформляется скриптами и ресурсами. `.tscn` отвечает главным образом за композицию и значения экспортируемых параметров.

### 8.3. Сигналы

Подключения сигналов должны быть локальными и понятными. Нельзя превращать один корневой узел в посредника для всех событий игры.

---

## 9. UI

UI может:

- читать состояние;
- отображать состояние;
- отправлять команды;
- показывать отклонение команды и причину.

UI не может:

- начислять монеты;
- наносить урон;
- самостоятельно менять роль;
- самостоятельно закреплять якорь;
- создавать игрового врага;
- решать условие поражения.

UI всегда считается представлением, а не владельцем игровой логики.

---

## 10. Тестируемость

Каждая система должна иметь интерфейс, позволяющий тестировать её без запуска полного забега.

Минимальные требования:

- чистая или изолируемая логика расчётов;
- отсутствие обязательной зависимости от HUD;
- настраиваемые данные;
- детерминированный режим случайности для тестов;
- сигналы результата;
- понятные причины отклонения команд.

При исправлении бага желательно сначала создать тест или воспроизводимый тестовый сценарий.

---

## 11. Изменение существующего кода

Перед изменением необходимо:

1. определить текущего владельца состояния;
2. проверить `PROJECT_RULES.md`;
3. проверить `ARCHITECTURE.md`;
4. оценить размер изменяемого файла;
5. убедиться, что изменение не добавляет вторую ответственность;
6. проверить, можно ли расширить существующую систему через компонент или данные.

Если файл уже близок к 450 строкам, новая функциональность должна добавляться через разделение, а не продолжение роста.

## 13. Запрещённые антипаттерны

- `Globals` или `GameManager`, содержащий всю игру.
- Файл свыше 600 строк.
- UI как владелец состояния.
- Несколько копий одного состояния.
- Прямое изменение чужих внутренних полей.
- Скрытые зависимости через глобальный поиск узлов.
- Хардкод баланса в логике.
- Большой `match` по типам всех сущностей проекта.
- Универсальный `Utils.gd` с несвязанными функциями.
- Сцена, содержащая всю игру без дочерних переиспользуемых сцен.
- Добавление механики вне архитектурного домена.
- Дублирование правил или архитектуры в нескольких документах.
