# GloryProtect — архитектура проекта

Этот файл — единственный источник правды для архитектуры. Обязательные ограничения находятся в [`PROJECT_RULES.md`](PROJECT_RULES.md).

## 1. Основные правила

- Godot 4.6.2 stable, строго типизированный GDScript.
- Максимум 600 строк на поддерживаемый файл; с 450 строк файл оценивается на разделение.
- Каждое изменяемое состояние имеет одного владельца.
- Настройки и баланс находятся в типизированных `Resource`.
- UI читает состояние и отправляет команды, но не реализует механику.
- Представление не изменяет симуляцию.
- Архитектурное изменение обновляет этот файл в том же наборе изменений.

## 5. PlatformDomain

`PlatformController` является единственным источником геометрии клеток:

```text
get_cell_count()
is_valid_cell(cell_index)
get_cell_local_x(cell_index)
get_nearest_cell_index(local_x)
```

BuildableDomain, роли, медицина, турели и визуал не повторяют формулу положения клетки.

## 6. BuildableDomain

```text
BuildableType
BuildableBalance
BuildableRuntime
BuildableSnapshot
BuildableInventory
BuildableGrid
BuildableDebugInput
BuildableGridVisual
```

Поддерживаемые типы:

```text
MEDICAL_STATION
TURRET
```

`BuildableInventory` владеет открытым количеством каждого типа. `BuildableGrid` владеет двумя структурами:

```text
Dictionary[buildable_id, BuildableRuntime]
Dictionary[cell_index, buildable_id]
```

Атомарные операции:

```text
place(type_id, cell_index)
move(buildable_id, cell_index)
demolish(buildable_id)
```

Правила:

- одна клетка содержит максимум один объект;
- служебные клетки недоступны;
- персонажи не блокируют установку;
- объекты не создают коллизий;
- размещённое количество не превышает открытое;
- демонтаж не удаляет открытие;
- наружу выдаются `BuildableSnapshot`.

`BuildableGridVisual` только рисует выбранную клетку, медицинский пост и турели.

## 7. ConcreteRoleStationDomain

Назначение защитника состоит из:

```text
current_role
current_station_id
target_role
target_station_id
state
```

Статические посты используют `station_id = -1`:

```text
DRIVER
LEFT_ANCHOR
RIGHT_ANCHOR
```

Медицинский пост использует один динамический `station_id = 0`.

Каждая турель использует:

```text
role_id = TURRET
station_id = buildable_id
```

`RoleStationRegistry` хранит владельца и координату по составному ключу:

```text
role_id : station_id
```

Поэтому две турели являются двумя независимыми постами одной роли.

Основной API:

```text
request_assignment(defender_id, role_id, station_id)
set_dynamic_role_station(role_id, available, local_x, station_id, relocate_active)
get_role_owner(role_id, station_id)
is_role_station_available(role_id, station_id)
```

Один пост не может иметь двух операторов. Один защитник не может одновременно занимать несколько постов.

## 8. Неделимые внешние действия

Системы ролей не знают устройство лечения или выстрела. Они получают только общий флаг:

```text
set_external_role_action_active(defender_id, role_id, active)
```

Пока флаг активен, обычное переназначение переходит в `WAITING_FOR_ACTION`. После снятия флага менеджер ролей начинает физический переход.

Сейчас этот интерфейс используют:

- `MedicalStationSystem` для пятисекундного цикла лечения;
- `TurretSystem` для уже начатого выстрела.

При смерти, демонтаже поста или принудительном освобождении роли флаг очищается.

## 9. MedicalStationDomain

`MedicalStationSystem` владеет целью и таймером лечения, но не здоровьем и не ролью.

Приоритет цели:

1. минимальное здоровье;
2. при равенстве — ближайший защитник.

Цикл:

```text
добежать до цели
→ 5 секунд непрерывного контакта
→ HealthComponent.heal(1)
→ повторная оценка целей
```

Активный лекарь не использует ближний бой. Перенос поста не прерывает цикл. Демонтаж немедленно освобождает роль.

## 10. TurretDomain

```text
TurretRuntime
TurretTargetSelector
TurretSystem
TurretDebugInput
```

### TurretRuntime

Каждая установленная турель имеет собственное состояние:

```text
buildable_id
operator_id
target_enemy_id
shot_remaining
cooldown_remaining
firing
```

Размещение и клетка не дублируются: они читаются из `BuildableGrid`.

### TurretTargetSelector

Получает реестр врагов и мировой центр турели. Возвращает ближайшую допустимую цель в круговом радиусе.

Допустимые цели:

```text
CLIMBING
ON_PLATFORM
FIGHTING
JUMPING
```

Наземные враги без пути не являются целями турели.

При равной дистанции выбирается враг с меньшим стабильным `enemy_id`.

### TurretSystem

`TurretSystem`:

- создаёт `TurretRuntime` на сигнал `buildable_placed`;
- регистрирует динамический пост `TURRET/buildable_id`;
- проверяет, что оператор жив, прибыл и роль активна;
- выбирает ближайшую цель;
- фиксирует её на весь выстрел;
- после `turret_shot_windup` вызывает `HealthComponent.apply_damage(1)`;
- запускает независимый кулдаун;
- не выдаёт монеты напрямую.

Смерть от турели проходит через существующую цепочку:

```text
HealthComponent.depleted
→ BoardingEnemy.kill("combat")
→ BoardingEnemyRegistry.enemy_removed
→ BoardingRewardController
→ RunEconomy / RunStatistics
```

### Одновременные выстрелы

Несколько турелей могут выбрать одного врага. Начатый выстрел не перенаправляется. Если цель уже погибла, выстрел завершается без попадания.

### Переназначение

Во время выстрела `TurretSystem` поднимает внешний флаг неделимого действия. Оператор завершает только текущий выстрел, затем покидает пост. Новый выстрел в `WAITING_FOR_ACTION` не начинается.

### Перенос

`BuildableGrid` мгновенно меняет клетку. Затем `CrewRoleManager`:

- обновляет координату динамического поста;
- перенаправляет уже идущего оператора;
- переводит активного оператора в физический переход;
- при активном выстреле ждёт его завершения.

Пока оператор не прибыл, турель не работает.

### Демонтаж

`TurretSystem`:

1. отменяет текущий выстрел;
2. снимает внешний флаг действия;
3. удаляет динамический пост;
4. освобождает оператора в `FREE_FIGHTER`;
5. удаляет `TurretRuntime`.

Открытие турели остаётся в `BuildableInventory`.

### Бой оператора

Роль `TURRET` не входит в список ролей, использующих ближний бой. Враг, дошедший до позиции турели, наносит урон оператору как обычному защитнику. У турели нет отдельного здоровья.

## 11. Balance

`BuildableBalance` является единственным источником параметров медицинского поста и турелей.

Текущая турель:

```text
max_count = 4
damage = 1
range = 360
shot_windup = 0.45
shot_cooldown = 0.8
```

## 12. Обязательные тестовые границы

### ConcreteRoleStationDomain

- две турели создают два независимых поста;
- один пост нельзя занять двумя защитниками;
- защитник может перейти с одной турели на другую;
- удаление конкретной турели не затрагивает другие посты;
- медицинский пост продолжает использовать собственный station id.

### TurretSystem

- закрытую турель нельзя установить;
- каждая открытая турель занимает отдельную клетку;
- без прибывшего оператора турель не стреляет;
- две турели независимо выбирают ближайшие цели;
- успешный выстрел наносит ровно один урон;
- убийство выдаёт обычную физическую награду;
- начатый выстрел завершается перед переназначением;
- новый выстрел при ожидающей команде не начинается;
- перенос во время выстрела ждёт его завершения;
- после переноса оператор физически прибывает к новой клетке;
- демонтаж немедленно освобождает оператора;
- открытие сохраняется после демонтажа;
- наземные враги без пути не являются целями.

### Общие

- здоровье изменяется только через `HealthComponent`;
- UI не владеет стрельбой или назначением;
- карточки и ручная пауза останавливают таймеры;
- лимит 600 строк не нарушен.

## 13. Следующая итерация

1. Визуальная анимация подготовки и выстрела турели.
2. Линия/снаряд без изменения боевой логики.
3. Отображение радиуса выбранной турели.
4. Визуальное состояние неработающей турели без оператора.
5. Подключение медицинского поста и турелей к будущему каталогу карточек.

Карточки остаются заглушками до отдельного проектирования каталога улучшений.

## 14. Запрещённые решения

- Второй владелец занятости клеток.
- Отдельная система ролей только для турелей.
- Один общий оператор для нескольких турелей.
- Хранение клетки турели внутри `TurretRuntime`.
- Прямое начисление монет из `TurretSystem`.
- Прямое удаление врага вместо `HealthComponent.apply_damage()`.
- Перенаправление уже начатого выстрела.
- Ближний бой активного турельщика.
- Работа турели до прибытия оператора.
- Продолжение работы после смерти оператора.
- Сохранение роли после демонтажа турели.
- Физическая коллизия турели с персонажами.
- Архитектурное изменение без обновления этого файла.
