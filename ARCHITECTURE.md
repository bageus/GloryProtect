# GloryProtect — архитектура проекта

Этот файл — единственный источник правды для архитектуры. Обязательные ограничения находятся в [`PROJECT_RULES.md`](PROJECT_RULES.md).

## 1. Основные правила

- Godot 4.6.2 stable, строго типизированный GDScript.
- Максимум 600 строк на поддерживаемый файл; с 450 строк файл оценивается на разделение.
- Каждое изменяемое состояние имеет одного владельца.
- Настройки и баланс находятся в типизированных `Resource`.
- UI читает состояние и отправляет команды, но не реализует механику.
- Представление не изменяет симуляцию.
- Архитектурное изменение обновляет этот файл в том же наборе изменений.

## 2. Направление зависимостей

Input / UI

## 5. GameFlowDomain

```text
BOOT
START_DELAY
RUNNING
CARD_SELECTION
MANUAL_PAUSE
GAME_OVER
```

`GameFlowController` владеет переходами, полной паузой и причиной поражения.

```text
start_run()
end_run(reason)
restart_run()
```

`restart_run()` разрешён только из `GAME_OVER` и перезагружает текущую сцену. Это гарантирует чистый щит, экипаж, платформу, врагов и таймеры без ручного частичного сброса.

## 6. DifficultyDomain

`RunDifficulty` является единственным владельцем активного времени забега. Время растёт только в `RUNNING`.

```text
get_elapsed_seconds()
get_normalized()
get_percent()
```

Стартовая задержка, карточки, ручная пауза и `GAME_OVER` время не увеличивают.

## 7. StatisticsDomain

```text
RunStatisticsSnapshot
RunStatistics
SessionRecordsStore
```

### RunStatistics

Владеет:

```text
_physical_kills
_final_snapshot
```

Время не дублируется: текущее и итоговое время читается из `RunDifficulty`.

Физическое убийство учитывается по сигналу:

```text
BoardingRewardController.reward_granted
```

Так используется тот же уже отфильтрованный набор физических смертей, который выдаёт монеты. Стратегические враги не попадают в этот поток.

При `run_ended` создаётся неизменяемый `RunStatisticsSnapshot`:

```text
survival_seconds
physical_kills
remaining_coins
purchased_upgrades
end_reason
```

### SessionRecordsStore

Узкое статическое хранилище текущего запуска приложения:

```text
completed_runs
best_survival_seconds
best_physical_kills
```

Оно переживает `reload_current_scene()`, но не записывает данные на диск и сбрасывается при перезапуске приложения. В нём запрещена любая игровая логика.

## 8. ResultsPresentation

### GameOverPanel

Показывает финальный снимок и рекорды текущего запуска:

- причину поражения;
- время выживания;
- физические убийства;
- оставшиеся монеты;
- число купленных карточек;
- количество завершённых забегов;
- лучшее время;
- максимум убийств.

Панель не изменяет статистику. Кнопка и `ui_accept` вызывают только `GameFlowController.restart_run()`.

### PrototypeHUD

Читает текущее время и убийства напрямую из `RunStatistics`. Копии результата в HUD запрещены.

## 9. ShieldCriticalAlertDomain

```text
ShieldSystem
    ↓ section_entered_critical
ShieldCriticalAlertController
    ↓ critical_alert_raised(section_ids)
ShieldCriticalAlertPresenter
```

`ShieldSystem` остаётся владельцем критического состояния каждой секции.

`ShieldCriticalAlertController`:

- принимает только переходы в критический диапазон;
- работает только в `RUNNING`;
- собирает секции, вошедшие в критическое состояние в одном кадре;
- выдаёт одно общее событие;
- очищает ожидающее событие при выходе из `RUNNING`.

`ShieldCriticalAlertPresenter`:

- показывает одно сообщение со списком секций;
- воспроизводит один короткий процедурный PCM-сигнал;
- не читает и не изменяет прочность щита;
- не повторяет звук, пока секция остаётся критической.

## 10. StrategicWaveDomain

```text
StrategicWaveBalance
StrategicWaveDirector
StrategicWaveSystem
StrategicGroupMutationController
StrategicGroupRuntime
StrategicGroupSnapshot
```

`StrategicWaveSystem` — единственный владелец массива групп. Группы существуют как агрегированные данные, не создаются как физические враги и не дают монет.

Поддерживаются:

- движение к секциям;
- отдельные импульсы урона от каждого врага;
- объединение близких масс;
- разделение на 2–3 части;
- перенаправление частей;
- сохранение общего количества врагов;
- плавное перестроение маршрутов без телепортации.

Миникарта получает только `StrategicGroupSnapshot`.

## 11. EconomyDomain и UpgradeDomain

`RunEconomy` — единственный владелец монет. Награды за физические смерти проходят через `BoardingRewardController`.

`UpgradeSystem` владеет выдачей карточек и числом покупок. Карточки пока одинаковы и не применяют эффект.

```text
5, 10, 15, …, 100, 200, 400, 800
```

## 12. Platform, Anchor, Crew и Boarding

- `PlatformController` владеет горизонтальной физикой.
- `WindSystem` владеет ветром.
- `AnchorRuntimeStore` владеет состояниями четырёх якорей.
- `CrewManager` владеет стабильными `defender_id`.
- `CrewRoleManager` владеет ролями и переходами.
- `BoardingEnemyRegistry` хранит только физических врагов.
- `BoardingSpawnDirector` читает `RunDifficulty`, но не хранит собственное время забега.

## 13. Обязательные тестовые границы

### Статистика и результаты

- Убийства учитываются только в `RUNNING`.
- Одна награждаемая физическая смерть добавляет одно убийство.
- Стратегические враги статистику убийств не изменяют.
- Финальный снимок фиксируется один раз при `run_ended`.
- После `GAME_OVER` результат не изменяется.
- Новый забег очищает текущую статистику.
- Лучшее время и максимум убийств сохраняются между перезагрузками сцены текущего запуска.
- Итоговый экран показывает снимок и рекорды.

### Критическое предупреждение

- Несколько секций в одном кадре дают один общий сигнал.
- Повторный сигнал появляется только после выхода и нового входа секции в критический диапазон.
- Во время карточек предупреждение не создаётся.
- Presenter получает готовый список и не читает состояние щита.

### Остальные системы

- Сложность растёт только в `RUNNING`.
- Карточки полностью останавливают симуляцию.
- Стратегические мутации сохраняют количество врагов.
- Без якорей физический спавн запрещён.
- Разрушение секции или всего экипажа завершает забег.
- Лимит 600 строк не нарушен.

## 14. Следующая итерация

1. Базовая система размещаемых объектов на клетках платформы.
2. Открытие медицинского поста без карточного эффекта — через debug-команду прототипа.
3. Физическое назначение лекаря к посту.
4. Лечение одного наиболее раненого защитника.
5. Подготовка общей основы для турелей.

Карточки остаются заглушками до отдельного проектирования каталога улучшений.

## 15. Запрещённые решения

- Второй владелец времени забега или физических убийств.
- Подсчёт убийств в HUD или итоговой панели.
- Запись стратегических врагов в физическую статистику.
- Полный игровой singleton под видом рекордов.
- Игровая логика внутри `SessionRecordsStore`.
- Перезапуск через частичный ручной сброс отдельных систем.
- Изменение щита или критического состояния из presenter.
- Отдельный звук для каждой секции одного кадра.
- UI как владелец экономики, статистики, сложности или симуляции.
- Архитектурное изменение без обновления этого файла.
