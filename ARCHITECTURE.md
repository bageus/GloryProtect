# GloryProtect — архитектура проекта

Этот файл является **единственным источником правды для архитектуры проекта**.

Обязательные инженерные ограничения находятся в [`PROJECT_RULES.md`](PROJECT_RULES.md).
Любое изменение владельцев состояния, границ систем или направления зависимостей должно обновлять этот файл в том же коммите.

---

## 1. Технологическая база

- Godot **4.6.2 stable**.
- GDScript.
- Композиция сцен и компонентов.
- Симуляция отделена от UI и визуального слоя.
- Настраиваемые параметры хранятся в типизированных `Resource`.
- Ни один исходный файл не превышает 600 строк.
- При приближении к 450 строкам файл оценивается на разделение.

---

## 2. Архитектурные правила

1. Один класс имеет одну ответственность.
2. Каждое изменяемое состояние имеет одного владельца.
3. UI отображает состояние и отправляет команды, но не реализует механику.
4. Доменные системы публикуют события после изменения состояния.
5. Зависимости передаются явно через сцену, `NodePath`, ресурс или небольшой интерфейс.
6. Балансные значения не хранятся в игровой логике.
7. Визуальный компонент не изменяет доменное состояние.
8. Новая механика добавляется в существующий домен либо требует обновления этого документа.
9. Временные прототипные инструменты не становятся владельцами игровых данных.

---

## 3. Направление зависимостей

```text
Input / UI
    ↓ команды
Domain Systems
    ↓ события и снимки состояния
UI / Visual / Audio
```

Доменные системы могут зависеть от:

- собственных ресурсов данных;
- явно переданных ссылок;
- небольших публичных интерфейсов соседних систем;
- событий верхнего уровня.

Доменные системы не зависят от:

- HUD;
- конкретных спрайтов;
- расположения панелей;
- глубоких путей чужих сцен;
- универсального глобального менеджера.

## 4. Высокоуровневая структура

```text
GameRoot
├── GameFlowDomain
├── DifficultyDomain
├── InputAdapters
├── World
│   ├── PlatformDomain
│   ├── AnchorDomain
│   ├── OrbDomain
│   ├── CrewDomain
│   ├── CombatDomain
│   ├── BoardingDomain
│   └── BuildableDomain
├── StrategicSimulation
│   ├── ShieldDomain
│   └── StrategicWaveDomain
├── EconomyDomain
└── PresentationDomain
```

`GameRoot` только собирает системы и передаёт зависимости. Он не реализует движение, бой, лечение, спавн, экономику или якорную физику.

---

## 5. Единственные владельцы состояния

| Состояние | Единственный владелец |
|---|---|
| Состояние забега и паузы | `GameFlowController` |
| Время активного забега | `RunClock` |
| Текущая сложность | `DifficultyDirector` |
| Направление и сила ветра | `WindSystem` |
| Позиция и скорость платформы | `PlatformController` |
| Доступность рулевого ввода | `SteeringInputProvider`, синхронизируемый `CrewRoleManager` |
| Состояния четырёх якорей | `AnchorRuntimeStore` |
| Очереди якорных операций | `AnchorOperationQueue` |
| Роли и переходы защитников | `CrewRoleManager` |
| Занятость и резервирование рабочих постов | `RoleStationRegistry` |
| Реестр защитников | `CrewManager` |
| Здоровье сущности | её `HealthComponent` |
| Контакт энергетических шаров | `OrbContactSystem` |
| Прочность пяти секций | `ShieldSystem` |
| Состояние физического врага | его state machine |
| Стратегические группы | `StrategicWaveSystem` |
| Занятость клеток объектами | `BuildableGrid` |
| Монеты забега | `RunEconomy` |
| Цена и выбор улучшений | `UpgradeSystem` |

UI и визуальные узлы могут хранить только восстанавливаемый кэш отображения.

## 6. GameFlowDomain

### GameFlowController

Владеет состояниями:

```text
BOOT
START_DELAY
RUNNING
CARD_SELECTION
MANUAL_PAUSE
GAME_OVER
```

Отвечает за:

- начало забега;
- стартовую задержку;
- полную ручную паузу;
- паузу выбора карточек;
- завершение забега;
- причину поражения.

Не отвечает за движение, урон, волны, роли или отображение паузы.

### RunClock

Планируемый компонент. Считает только активное время забега и останавливается при полной паузе.

## 7. PlatformDomain

Текущая композиция:

```text
Platform
├── PlatformController
├── PlatformVisualController
├── CrewManager
├── CrewRoleManager
└── Camera2D

GameRoot
├── SteeringInputProvider
└── WindSystem
```

### PlatformController

Единственный владелец:

- горизонтальной позиции;
- горизонтальной скорости;
- применения сил;
- сопротивления;
- ограничения скорости;
- мировых и якорных границ.

Получает:

- рулевую ось от `SteeringInputProvider`;
- усилие ветра от `WindSystem`;
- диапазон движения от `AnchorSystem`;
- параметры из `PlatformBalance`.

Не читает клавиатуру и не рисует платформу.

### SteeringInputProvider

Преобразует действия ввода в рулевую ось.

Правила:

- без активного рулевого возвращает 0;
- сообщает, удерживается ли хотя бы одна кнопка направления;
- одновременное удержание двух направлений считается активным вводом с нулевой осью;
- `CrewRoleManager` включает его только после прибытия рулевого.

### WindSystem

Единственный владелец:

- направления;
- уровня силы 1–3;
- фактического усилия;
- времени следующего изменения.

Использует `WindBalance` и не двигает платформу напрямую.

### PlatformVisualController

Рисует временную платформу, клетки, посты и шар. Не изменяет движение, роли или состояние экипажа.

### Ресурсы

```text
PlatformBalance
WindBalance
```

`PlatformBalance` является источником правды для размеров платформы, расположения постоянных постов и параметров движения.

---

## 8. AnchorDomain

Текущая композиция:

```text
AnchorSystem
├── AnchorRuntimeStore
├── AnchorCommandController
├── AnchorOperationQueue
├── AnchorConstraintProvider
├── AnchorOverloadController
├── AnchorGeometry
└── AnchorVisualController
```

### AnchorSystem

Тонкий координатор. Перенаправляет команды, собирает компоненты и повторно публикует события.

### AnchorRuntimeStore

Единственный владелец состояний:

```text
STOWED
QUEUED
INSTALLING
ATTACHED
OVERLOADED
RETURNING
```

### AnchorCommandController

Проверяет и маршрутизирует:

```text
request_toggle(anchor_id)
request_remove_all()
operator_availability_changed(side, available)
```

### AnchorOperationQueue

Владеет текущей установкой, очередью каждой стороны и ожиданием экстренного снятия.

Начатая установка завершается после потери оператора. Ещё не начатые команды отменяются при отключении поста.

### AnchorGeometry

Вычисляет:

- наземные точки;
- точки платформы;
- конечную длину троса;
- геометрические границы движения.

### AnchorConstraintProvider

Предоставляет платформе допустимый диапазон или полную фиксацию. Не изменяет позицию платформы.

### AnchorOverloadController

Владеет таймером перегрузки. Использует реальное натяжение и направление ветра.

### AnchorVisualController

Рисует силуэты, тросы, перегрузку и возврат. Не меняет runtime.

### AnchorBalance

Единственный источник настраиваемых параметров якорей.

## 9. CrewDomain

Текущая композиция:

```text
CrewManager
├── Defender 1
├── Defender 2
└── Defender 3

CrewRoleManager
└── RoleStationRegistry

Defender
├── HealthComponent
├── DefenderMovement
└── DefenderVisual
```

### CrewManager

Единственный владелец реестра защитников.

Текущие обязанности:

- создать стартовый экипаж;
- выдать уникальные идентификаторы;
- предоставить доступ к защитникам;
- сообщить о смерти.

Позднее сюда добавляются независимые таймеры замены и выход нового защитника из двери.

### CrewRoleManager

Единственный владелец:

- текущей роли;
- целевой роли;
- состояния перехода;
- ожидания завершения неделимого действия.

Состояния назначения:

```text
ACTIVE
WAITING_FOR_ACTION
MOVING
DEAD
```

Подтверждённое поведение:

- рулевой уходит только после отпускания обеих кнопок;
- якорщик уходит после завершения текущей установки;
- новая роль активируется только после физического прибытия;
- во время перехода старый пост уже отключён, новый ещё не работает.

### RoleStationRegistry

Единственный владелец занятости и резервирования постоянных постов.

Предоставляет:

```text
reserve(role_id, defender_id)
release(role_id, defender_id)
get_owner(role_id)
get_target_x(role_id, defender_id)
```

Геометрия постов берётся из `PlatformBalance`, а не дублируется в `CrewBalance`.

### Defender

Тонкий корень композиции. Не владеет ролью.

### DefenderMovement

Владеет текущей целью движения конкретного защитника и сообщает о прибытии.

### HealthComponent

Единственный владелец здоровья сущности.

### DefenderVisual

Рисует временный корпус и сегменты здоровья. Не изменяет здоровье или роль.

### CrewDebugInput

Временный адаптер ввода прототипа. Отправляет команды в `CrewRoleManager` и не хранит назначения.

### CrewBalance

Содержит количество стартовых защитников, здоровье, скорость движения и временные визуальные размеры.

## 10. CombatDomain

Текущая реализация содержит универсальный `HealthComponent`.

Целевая композиция:

```text
HealthComponent
TargetSelector
MeleeAttackComponent
RangedAttackComponent
CombatResolver
DeathComponent
```

Правила:

- здоровье меняется только через `HealthComponent`;
- завершённые попадания применяет `CombatResolver`;
- начатая атака не перенаправляется;
- визуал не наносит урон;
- выбор цели отделён от выполнения атаки.

---

## 11. OrbDomain и ShieldDomain

### GroundOrbRegistry

Будущий источник пяти определений шаров: позиция, секция, контактная зона, якорная зона и четыре крепления.

### OrbContactSystem

Будущий владелец текущего контакта. Публикует:

```text
contact_started(orb_id)
contact_ended(orb_id)
```

### ShieldSystem

Будущий единственный владелец пяти секций.

Принимает:

```text
apply_damage(section_id, amount)
restore(section_id, amount)
```

### ShieldRechargeController

Преобразует активный контакт в команды восстановления `ShieldSystem`.

Текущие `PrototypeWorld` и `PrototypeShieldSystem` являются временными компонентами и должны быть заменены этим доменом.

---

## 12. BoardingDomain

Целевая композиция:

```text
BoardingSpawnDirector
GroundEnemyRegistry
AnchorPathRegistry
BoardingEnemy
└── BoardingEnemyStateMachine
```

`BoardingSpawnDirector` владеет интервалом спавна, общим наземным лимитом и стороной появления.

`AnchorPathRegistry` предоставляет доступные маршруты. Враг фиксирует выбранный маршрут до его недоступности.

Состояния врага:

```text
SPAWNING_OFFSCREEN
RUNNING_TO_ANCHOR
WAITING_WITHOUT_PATH
CLIMBING
ENTERING_PLATFORM
FIGHTING
JUMPING
DEAD
```

---

## 13. StrategicWaveDomain

Стратегические враги представлены агрегированными данными, а не отдельными сценами.

```text
StrategicWaveDirector
StrategicWaveSystem
MinimapRenderer
```

`StrategicWaveSystem` владеет группами, их количеством, целью, движением, объединением, разделением и отдельными импульсами урона.

`MinimapRenderer` только отображает данные.

---

## 14. BuildableDomain

Целевая композиция:

```text
BuildableInventory
BuildableGrid
MedicalStationSystem
TurretSystem
```

`BuildableGrid` является единственным владельцем занятости клеток объектами.

Медицинский пост и турели используют обычные свободные клетки и не блокируют персонажей.

---

## 15. EconomyDomain

### RunEconomy

Единственный владелец монет текущего забега.

### UpgradeSystem

Владеет номером выдачи, стоимостью, текущими двумя карточками и последовательной цепочкой покупок.

### UpgradeCatalog

Хранит data-driven определения. Эффекты не реализуются одним большим `match`.

---

## 16. PresentationDomain

Содержит:

- HUD;
- миникарту;
- указатели;
- визуалы платформы, якорей и персонажей;
- анимации;
- звук;
- эффекты.

Presentation:

- читает публичные снимки;
- подписывается на события;
- отправляет команды;
- не меняет доменное состояние напрямую.

## 19. Тестовые границы

Минимальные обязательные сценарии:

- конечная длина троса в обоих направлениях;
- перегрузка одиночного натянутого якоря;
- устойчивость двух якорей стороны;
- рулевой не уходит при активном вводе;
- управление отключено во время перехода;
- управление включается после прибытия нового рулевого;
- якорщик завершает текущую установку перед уходом;
- параметры платформы и ветра читаются из ресурсов;
- ни один поддерживаемый файл не превышает 600 строк.

---

## 20. Следующая архитектурная итерация

1. заменить тестовую секцию полноценным `ShieldSystem`;
2. добавить `GroundOrbRegistry` и `OrbContactSystem`;
3. добавить `DefenderActionController`;
4. реализовать базовый ближний бой;
5. добавить смерть всего экипажа как условие поражения;
6. добавить независимые таймеры замены;
7. добавить базового физического врага и его state machine.

---

## 21. Запрещённые решения

- Глобальный объект, управляющий всей игрой.
- UI как владелец игровых данных.
- Дублирование роли в защитнике и `CrewRoleManager`.
- Дублирование геометрии постов в нескольких ресурсах.
- Зависимость симуляции от визуала.
- Один огромный скрипт для ролей, врагов или карточек.
- Балансные значения, разбросанные по логике.
- Глубокие пути между независимыми сценами.
- Физическая сцена на каждого стратегического врага.
- Архитектурное изменение без обновления этого файла.
