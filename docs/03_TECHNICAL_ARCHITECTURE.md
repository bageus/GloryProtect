# GloryProtect — техническая архитектура

Этот документ описывает рекомендуемое разделение систем для реализации проекта в Godot 4. Архитектура ориентирована на модульность, независимое тестирование механик и отсутствие одного глобального контроллера, содержащего всю логику игры.

---

# 1. Архитектурные принципы

## 1.1. Разделение симуляции и отображения

Игровые правила не должны зависеть от конкретных UI-узлов или анимаций.

Пример:

- `ShieldSystem` изменяет прочность секции;
- система отправляет сигнал;
- `ShieldPanel` обновляет процент и цвет;
- `GroundOrbView` меняет яркость;
- `DirectionIndicatorController` показывает или скрывает стрелку.

Ни один UI-компонент не должен самостоятельно изменять состояние щита.

## 1.2. Конфигурация через данные

Все настраиваемые значения должны храниться в ресурсах или конфигурационных объектах:

- скорость платформы;
- силы ветра;
- длина тросов;
- длительность установки;
- время перегрузки;
- урон;
- кулдауны атак;
- скорость лечения;
- параметры волн;
- рост сложности;
- стоимость улучшений.

Числа не должны быть разбросаны по скриптам.

## 1.3. Событийная связь

Системы должны общаться через сигналы и команды, а не через произвольные глубокие вызовы `get_node()`.

Рекомендуется использовать:

- локальные сигналы узлов;
- центральную шину событий только для событий уровня забега;
- прямые типизированные ссылки между родителем и дочерними узлами;
- интерфейсы или небольшие компоненты вместо проверки конкретных классов.

## 1.4. Машины состояний

Машины состояний нужны для:

- общего потока игры;
- защитников;
- физических врагов;
- якорей;
- стратегических групп;
- размещаемых объектов при необходимости.

Состояния не должны храниться в десятках независимых булевых переменных.

## 1.5. Один источник истины

Каждое состояние хранится только в одной системе.

Примеры:

- прочность секций — только в `ShieldSystem`;
- назначение ролей — только в `CrewRoleManager`;
- состояние якоря — только в `AnchorSystem`;
- монеты — только в `RunEconomy`;
- состояние паузы — только в `GameFlowController`.

# 4. GameFlowController

## 4.1. Ответственность

- запуск забега;
- стартовая безопасная пауза;
- переход в активную игру;
- ручная пауза;
- пауза выбора карточек;
- завершение забега;
- причина поражения;
- блокировка ввода в зависимости от состояния.

## 4.2. Состояния

```gdscript
enum RunState {
    BOOT,
    START_DELAY,
    RUNNING,
    CARD_SELECTION,
    MANUAL_PAUSE,
    GAME_OVER,
}
```

## 4.3. Правила паузы

`GameFlowController` должен централизованно управлять паузой дерева либо собственным симуляционным масштабом.

Во время `CARD_SELECTION` и `MANUAL_PAUSE` останавливаются:

- физика платформы;
- ветер;
- стратегические волны;
- физические враги;
- атаки;
- лечение;
- таймеры восстановления;
- зарядка щита.

В `MANUAL_PAUSE` разрешены только меню и настройки.

# 5. DifficultyDirector

## 5.1. Назначение

`DifficultyDirector` хранит единый нормализованный или абсолютный параметр сложности, зависящий от времени забега.

Он не создаёт врагов напрямую, а выдаёт текущие параметры двум независимым системам:

```text
DifficultyDirector
├── StrategicWaveDirector
└── BoardingSpawnDirector
```

## 5.2. Пример данных

```gdscript
class_name DifficultySnapshot

var elapsed_time: float
var strategic_wave_size: int
var strategic_spawn_interval: float
var strategic_max_active_groups: int
var boarding_spawn_interval: float
var boarding_ground_limit: int
```

## 5.3. Требование

Рост должен быть плавным. Не следует жёстко привязывать его только к дискретным уровням или завершённым волнам.

---

# 6. PlatformMovementSystem

## 6.1. Ответственность

- горизонтальная скорость;
- управляющая сила;
- воздействие ветра;
- инерция;
- трение или сопротивление;
- ограничения тросами;
- границы уровня;
- обнуление скорости при полной блокировке.

## 6.2. Входные данные

```gdscript
class_name PlatformMovementInput

var steering_axis: float
var steering_available: bool
var wind_force: float
var left_constraint: AnchorConstraint
var right_constraint: AnchorConstraint
```

## 6.3. Важные правила

- Ввод не должен действовать без рулевого.
- Якорная блокировка не должна накапливать скрытый импульс.
- При достижении максимальной длины троса скорость в блокируемом направлении сразу обнуляется.
- Платформа не раскачивается как маятник.
- Границы уровня работают как жёсткие ограничения.

# 7. WindSystem

## 7.1. Состояние

```gdscript
class_name WindState

var direction: int # -1 или 1
var strength_level: int # 1, 2, 3
var actual_force: float
var next_change_time: float
```

## 7.2. Ответственность

- выбор направления;
- выбор уровня силы;
- небольшие колебания фактической силы;
- отправка событий при изменении уровня;
- предоставление текущей силы движению платформы;
- уведомление якорной системы о ветре силы 3.

`WindSystem` не должен напрямую менять состояние якоря.

# 8. ShieldSystem

## 8.1. Модель секции

```gdscript
class_name ShieldSectionState

var section_id: int
var current_health: float
var max_health: float = 100.0
var linked_orb_id: int
var is_critical: bool
var needs_direction_indicator: bool
```

## 8.2. Ответственность

- хранение пяти секций;
- применение отдельных импульсов урона;
- восстановление;
- проверка 50%-порога;
- проверка критического порога;
- проверка 0%;
- отправка событий интерфейсу и `GameFlowController`.

## 8.3. Сигналы

```gdscript
signal section_changed(section_id: int, current_health: float)
signal section_indicator_changed(section_id: int, visible: bool)
signal section_entered_critical(section_id: int)
signal section_destroyed(section_id: int)
```

Общий предупреждающий звук должен управляться отдельным `CriticalAlertController`, чтобы несколько одновременных событий не создавали звуковой спам.

# 9. OrbSystem

## 9.1. GroundOrb

Каждый наземный шар хранит:

- идентификатор;
- связанную секцию;
- узкую зону контакта;
- расширенную зону креплений;
- четыре точки крепления;
- визуальное состояние.

## 9.2. OrbContactController

Отдельный компонент определяет, находится ли шар платформы внутри зоны контакта.

Он должен отправлять события:

```gdscript
signal contact_started(orb_id: int)
signal contact_ended(orb_id: int)
```

Сам компонент не меняет прочность щита. Зарядкой занимается `ShieldRechargeController`, получающий текущий контакт.

# 10. AnchorSystem

## 10.1. Модель якоря

```gdscript
enum AnchorState {
    STOWED,
    INSTALL_QUEUED,
    INSTALLING,
    ATTACHED,
    OVERLOADED,
    REMOVING,
    BROKEN_RETURNING,
}
```

```gdscript
class_name AnchorRuntime

var anchor_id: int
var side: int
var state: AnchorState
var attachment_point: Node2D
var rope_length: float
var overload_progress: float
var assigned_operator_id: int
```

## 10.2. Компоненты

Рекомендуется разделить:

- `AnchorCommandController` — принимает команды игрока;
- `AnchorOperationQueue` — хранит максимум допустимых операций стороны;
- `AnchorConstraintProvider` — сообщает ограничения движению;
- `AnchorOverloadController` — управляет ветровой перегрузкой;
- `AnchorBoardingPath` — предоставляет врагам маршрут подъёма;
- `AnchorVisualController` — трос, мигание, силуэты.

## 10.3. Неделимые операции

Установка якоря оформляется как объект операции:

```gdscript
class_name AnchorOperation

var type: int
var anchor_id: int
var progress: float
var is_started: bool
var is_completed: bool
```

После запуска операция должна завершиться даже при смерти оператора. Поэтому прогресс операции нельзя хранить только внутри `Defender`.

## 10.4. Снятие всех якорей

Команда должна создавать операции снятия только на обслуживаемых сторонах и отменять только ещё не начатые установки.

# 11. CrewManager и роли

## 11.1. CrewManager

Хранит:

- список защитников;
- живое или мёртвое состояние;
- текущее здоровье;
- таймеры замены;
- доступность портретов;
- идентификаторы назначенных ролей.

Он не должен реализовывать конкретное поведение лекаря, рулевого или турельщика.

## 11.2. CrewRoleManager

Отвечает за:

- проверку доступности поста;
- резервирование роли;
- снятие предыдущего назначения;
- движение к новой позиции;
- активацию роли после прибытия;
- освобождение роли при смерти или демонтаже объекта.

## 11.3. Роль как данные

```gdscript
class_name RoleDefinition extends Resource

@export var role_id: StringName
@export var station_type: StringName
@export var allows_melee: bool
@export var movement_target_offset: Vector2
```

Конкретные роли не должны требовать большого `match` внутри одного контроллера.

---

# 12. DefenderStateMachine

Рекомендуемые состояния:

```gdscript
enum DefenderState {
    IDLE,
    MOVING_TO_ROLE,
    FREE_FIGHTING,
    STATION_FIGHTING,
    DRIVING,
    OPERATING_ANCHOR,
    OPERATING_TURRET,
    HEALING,
    WAITING_AT_STATION,
    DEAD,
}
```

## 12.1. Неделимое действие

У защитника должен быть общий интерфейс текущего действия:

```gdscript
class_name AtomicAction

func can_interrupt() -> bool:
    return false

func tick(delta: float) -> void:
    pass

func is_finished() -> bool:
    return false
```

Новая роль помещается в `pending_assignment`, пока текущее действие не завершено.

## 12.2. Исключения

- Установка якоря должна продолжаться в самой `AnchorSystem`, даже если защитник погиб.
- Рулевой заканчивает действие после отпускания обеих кнопок.
- Лекарь заканчивает текущий полный цикл лечения.
- Турельщик заканчивает текущий выстрел.
- Боец заканчивает текущий взмах.

# 13. CombatSystem

## 13.1. Компоненты

- `HealthComponent`;
- `TargetSelector`;
- `MeleeAttackComponent`;
- `RangedAttackComponent`;
- `DamageReceiver`;
- `DeathComponent`.

## 13.2. Урон

Базовый урон обычного врага, меча и турели равен 1.

Система должна позволять будущие типы врагов с другим здоровьем или скоростью, не меняя правило базового удара.

## 13.3. Фиксация цели атаки

При запуске атаки сохраняется ссылка или стабильный идентификатор цели.

Если цель погибла до момента попадания:

- атака завершается;
- урон не наносится;
- новая цель не подставляется автоматически.

## 13.4. Одновременные атаки

У каждого атакующего собственный кулдаун. Центральной очереди атак быть не должно.

---

# 14. BoardingEnemySystem

## 14.1. Состояния врага

```gdscript
enum BoardingEnemyState {
    SPAWNING_OFFSCREEN,
    RUNNING_TO_ANCHOR,
    WAITING_WITHOUT_PATH,
    CLIMBING,
    ENTERING_PLATFORM,
    FIGHTING,
    JUMPING,
    DEAD,
}
```

## 14.2. Выбор якоря

Алгоритм:

1. получить список доступных якорей;
2. найти минимальную дистанцию по земле;
3. собрать якоря в допустимом диапазоне равенства;
4. при одном кандидате выбрать его;
5. при нескольких выбрать случайно;
6. зафиксировать цель до её недоступности.

## 14.3. Отсутствие пути

При отсутствии доступных якорей враг переходит в `WAITING_WITHOUT_PATH` и продолжает двигаться вместе с платформой по земле, но не способен наносить урон.

## 14.4. Общий лимит

Лимит хранится в `BoardingSpawnDirector`, а не отдельно для левой и правой стороны.

## 14.5. Подъём и пробки

Каждый якорь предоставляет отдельный одномерный маршрут. Враги имеют минимальную дистанцию друг до друга и не могут проходить сквозь союзников.

# 15. JumpOverDefenderController

Прыжок разрешается только при выполнении всех условий:

- враг заблокирован союзным врагом впереди;
- между врагом и защитником нет другого свободного пути;
- за защитником есть свободная посадочная позиция;
- посадочная позиция не занята другим врагом.

Во время прыжка:

- коллизия движения может быть временно изменена;
- `HealthComponent` остаётся активным;
- турели могут продолжать выбирать врага целью.

После приземления враг немедленно запускает обычный выбор ближайшего защитника.

---

# 16. StrategicWaveSystem

## 16.1. Представление групп

Не рекомендуется создавать физический узел для каждого стратегического врага.

```gdscript
class_name StrategicGroupRuntime

var group_id: int
var enemy_count: int
var normalized_position: float
var target_section_id: int
var speed: float
var damage_per_enemy: float
var impact_rate: float
```

## 16.2. Спавн волн

`StrategicWaveDirector` создаёт одну или несколько групп на основе случайного распределения целей.

## 16.3. Объединение

При выполнении условий близости и случайной проверки две группы могут объединиться:

- количество складывается;
- позиция выбирается по согласованному правилу;
- цель должна быть совместимой;
- визуальный размер пересчитывается.

## 16.4. Разделение

Группа может случайно разделиться в любой момент:

- сумма врагов дочерних групп равна исходному количеству;
- дочерние группы могут получить разные цели;
- ни один враг не должен теряться или создаваться из-за операции.

## 16.5. Столкновение со щитом

Группа переходит в состояние воздействия и с заданным темпом выдаёт отдельные импульсы:

```gdscript
shield_system.apply_damage(target_section_id, damage_per_enemy)
enemy_count -= 1
```

После `enemy_count == 0` группа удаляется.

# 17. MedicalSystem

## 17.1. MedicalStation

Хранит:

- установленное или резервное состояние;
- текущую клетку;
- назначенного лекаря;
- рабочую доступность.

## 17.2. Выбор пациента

Сортировка целей:

1. меньшее текущее здоровье;
2. меньшее расстояние до лекаря;
3. стабильный идентификатор как последний детерминированный критерий.

## 17.3. Лечебный цикл

Цель фиксируется на полный цикл. После завершения восстанавливается один сегмент и выполняется новый поиск.

Если цель погибла до прибытия, выбор отменяется сразу.

Если пост перемещён, текущий цикл завершается, после чего лекарь идёт к новой позиции.

---

# 18. TurretSystem

## 18.1. TurretRuntime

```gdscript
class_name TurretRuntime

var turret_id: int
var grid_cell: int
var operator_id: int
var is_operational: bool
var attack_cooldown: float
var range: float
```

## 18.2. Выбор цели

Турель рассматривает:

- врагов на платформе;
- врагов на якорях;
- прыгающих врагов.

Выбирается ближайшая живая цель внутри кругового радиуса.

## 18.3. Перемещение

При переносе:

- клетка турели меняется мгновенно;
- `is_operational = false`;
- оператор получает новый маршрут;
- после прибытия работа возобновляется.

## 18.4. Демонтаж

При демонтаже:

- клетка освобождается;
- объект возвращается в резерв;
- оператор освобождается и становится свободным бойцом.

---

# 19. BuildableObjectManager

Отвечает за:

- свободные и занятые ячейки;
- резерв неустановленных объектов;
- установку;
- перенос;
- демонтаж;
- ограничение одного медицинского поста;
- связь объектов с операторами.

Проверка занятости касается только других размещаемых объектов. Наличие персонажа на клетке не запрещает установку.

---

# 20. RunEconomy и UpgradeSystem

## 20.1. RunEconomy

Хранит только:

- текущие монеты;
- общее количество заработанных монет;
- события изменения баланса.

Монеты добавляются по событию смерти физического врага с допустимой причиной.

```gdscript
enum EnemyDeathCause {
    DEFENDER,
    TURRET,
    ANCHOR_REMOVAL,
    ANCHOR_BREAK,
    OTHER,
}
```

`OTHER` по умолчанию не обязан давать монеты.

## 20.2. UpgradeSystem

Хранит:

- номер следующей выдачи;
- текущую стоимость;
- пул доступных определений;
- текущие две карточки;
- состояние последовательных покупок.

Формула:

```gdscript
func get_upgrade_cost(index: int) -> int:
    if index <= 20:
        return index * 5
    return 100 * int(pow(2, index - 20))
```

Индекс первой выдачи равен 1.

## 20.3. Data-driven карточки

```gdscript
class_name UpgradeDefinition extends Resource

@export var id: StringName
@export var title: String
@export_multiline var description: String
@export var icon: Texture2D
@export var tags: Array[StringName]
@export var max_level: int
@export var weight: float
```

Эффект карточки должен реализовываться отдельной стратегией или компонентом, а не большим `match` по ID.

# 23. Порядок обновления симуляции

Рекомендуемый порядок одного физического тика:

1. проверить состояние паузы;
2. обновить ветер;
3. обработать рулевой ввод;
4. получить якорные ограничения;
5. обновить движение платформы;
6. обновить положение наземных врагов относительно платформы;
7. обновить подъём по якорям;
8. обновить перемещение защитников и врагов на платформе;
9. обновить атаки и лечение;
10. применить завершённые действия;
11. обновить контакт шаров и зарядку;
12. проверить условия поражения;
13. отправить агрегированные события UI.

Стратегическая симуляция может обновляться с меньшей фиксированной частотой, чем физический мир, если движение миникарты остаётся визуально плавным.

# 24. Тестируемость

Минимальные автоматические тесты:

## 24.1. Щит

- секция достигает 0% и создаёт поражение;
- указатель появляется при 50%;
- исчезает выше 50%;
- зарядка не превышает 100%.

## 24.2. Якоря

- одиночный якорь не срывается при ветре 2;
- перегружается при ветре 3;
- второй якорь отменяет перегрузку;
- ослабление ветра сбрасывает таймер;
- недостаточная длина вызывает немедленный обрыв;
- снятие убивает только врагов на тросе.

## 24.3. Бой

- базовый враг погибает от одного удара;
- несколько врагов независимо атакуют одну цель;
- лишняя начатая атака не перенаправляется;
- поражение создаётся при отсутствии живого экипажа.

## 24.4. Лечение

- лечится самый тяжело раненый;
- при равенстве выбирается ближайший;
- цель фиксируется на цикл;
- смерть цели отменяет маршрут;
- перенос поста ждёт завершения цикла.

## 24.5. Экономика

- стратегические враги не дают монет;
- физические допустимые смерти дают монеты;
- стоимость 20-й выдачи равна 100;
- 21-й — 200;
- последовательные покупки продолжаются до нехватки монет.

# 25. Антипаттерны, которых следует избегать

- Один глобальный скрипт со всеми системами.
- Прямое изменение UI из боевой логики.
- Проверки ролей через десятки строк `if role == ...` в каждом классе.
- Хранение состояния якоря одновременно в платформе, враге и UI.
- Физический узел на каждого врага стратегической волны.
- Жёстко зашитые числовые параметры.
- Отмена начатых действий при любом новом вводе.
- Случайный выбор якоря без фиксации цели.
- Принудительное выравнивание левого и правого спавна после случайного перекоса.
- Использование прочности щита как прямого множителя физического абордажа.
